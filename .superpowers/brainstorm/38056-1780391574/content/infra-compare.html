SSE Infrastructure: Option A vs Option C

เปรียบเทียบ architecture เมื่อ scale เป็น multi-instance

Option A — In-memory EventEmitter
Single instance (ปัจจุบัน) ✓
Browser (EventSource)
↕SSE
Node.js Process
in-memory Map<shopId, Set<Writer>>
↕
PostgreSQL
Multi-instance (scale out) ✗
Browser A
↕
Server 1
emitter (memory)
Browser B
↕
Server 2
emitter (memory)
PostgreSQL
ปัญหา: User A เชื่อม Server 1 อยู่
Mutation เกิดที่ Server 2
→ emitter ของ Server 1 ไม่รู้
→ User A ไม่ได้รับ event
แก้ได้ด้วย: Sticky session (nginx ip_hash) — ผูก user กับ server เดิมเสมอ แต่ทำให้ load balance ไม่สม่ำเสมอ
Option C — Redis Pub/Sub
Single instance (ปัจจุบัน) ✓
Browser (EventSource)
↕SSE
Node.js Process
subscribe Redis channel
↕
PostgreSQL
Redis
Multi-instance (scale out) ✓
Browser A
↕
Server 1
sub: shop:ABC
Browser B
↕
Server 2
sub: shop:ABC
↑ pub: shop:ABC ↑
PostgreSQL
Redis
message bus
ทำงาน: Mutation เกิดที่ Server ไหนก็ได้
publish ไปที่ Redis → Redis broadcast ให้
ทุก Server ที่ subscribe → ทุก Browser ได้รับ
เพิ่มใน docker-compose: redis service 1 container
เพิ่ม dependency: ioredis (~45KB)
เพิ่ม logic: publish หลัง mutation, subscribe ใน /api/events
สรุป: ถ้าแผนคือ scale ออกแน่นอน → Option C ไม่ใช่ overkill แต่เป็น ลงทุนถูกที่ ค่าใช้จ่ายคือ Redis container + ioredis + ~30 บรรทัด ประหยัดกว่า refactor ทีหลังมาก
